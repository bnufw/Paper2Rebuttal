# ICML/OpenReview 模式 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 增加 ICML 2026 / OpenReview author response 工作流：按“每个 official review 单独一条”生成可直接粘贴的 rebuttal/后续 follow-up，严格不超过 `5000 characters`，并加入匿名与合规护栏；严禁编造实验结果，缺失证据用占位符由作者后续补全。

**Architecture:** 在现有 multi-agent pipeline 基础上，新增“OpenReview 结构化抽取 -> per-reviewer 聚合 -> 生成 -> 合规扫描/重写 -> 字符预算压缩”的后处理链路；同时提供 follow-up 模式（reviewer comment 后的短回复草稿 + 最小证据清单）。

**Tech Stack:** Python, Gradio (`app.py`), OpenAI-compatible client (`llm.py`), YAML prompts (`prompts/*.yaml`), 现有 session/log 机制 (`rebuttal_service.py`, `tools.py`)。

## 约束与事实确认（ICML 2026）

- 讨论包含三轮（rebuttal、reviewer follow-up、author follow-up），每轮每条消息限制 `5000 characters`。来源：ICML 2026 Reviewer Instructions。citeturn0search0
- 作者回复需保持双盲：不得包含可能泄露身份的信息；不得包含非匿名 URL、个人网站 URL、短链（tinyurl 等）；reviewer 不应被期望点击外链。来源：ICML 2026 Author Instructions。citeturn0search2
- rebuttal 期不能上传修订稿；只能在回复中澄清/解释。来源：ICML 2026 Author Instructions。citeturn0search2

## 现状差距（基于仓库代码）

- `prompts/rebuttal_writer.yaml` / `prompts/rebuttal_reviewer.yaml` 以 “CVPR rebuttal formatting” 为目标，且要求缺实验时“编造数值并加 *”，不符合你要求的“严禁编造、用占位符”。（需要替换为 ICML/OpenReview 风格。）
- `rebuttal_service.py:generate_final_rebuttal()` 只生成一份“总 rebuttal”，不支持 per-reviewer 单条、也没有 `5000 characters` 预算与压缩链路。
- 当前未做匿名/合规扫描：URL、短链、个人信息、以及“我们已修改论文”等不符合 rebuttal 期事实的时态表达无法自动拦截。
- follow-up 回合未建模：缺少“reviewer follow-up comment -> author follow-up reply（短、5000 chars）+ 最小证据清单”的专用产物与 UI。

---

## 设计决策（你已确认的需求）

1. **输出粒度**：按 OpenReview “每个 official review 单独一条”生成；每条可直接粘贴，严格 `<= 5000 characters`（含换行与空格）。
2. **合规护栏**：自动扫描并拒绝（或触发自动重写）以下内容：
   - `http(s)://`、`www.`、短链域、个人网站/非匿名 URL。
   - 可能泄露身份的个人信息（邮箱、电话、真实姓名片段等，先做 conservative 规则）。
   - 明显不合时态的表达（例如 “we have updated the paper”），统一改为 “we will incorporate in the camera-ready / subsequent version”。
3. **不编造实验**：提示词与后处理都必须禁止“生成新数字/结果”。缺失结果时用统一占位符，例如 `[[TBD: ablation on X]]`、`[[TBD: figure showing Y]]`。
4. **支持后续讨论回合**：生成 reviewer follow-up 的短回复草稿（同 `5000 characters`），并附带“你需要补充什么证据/截图/数值”的最小需求列表（同样用 `[[TBD: ...]]` 占位）。

---

## 统一输出规范（建议）

- **模板（rebuttal）**：`Response to <ReviewerID>` 开头，随后按该 reviewer 的问题编号（Q1/Q2...）逐条回复。
- **模板（follow-up）**：`Reply to <ReviewerID> follow-up` 开头，最多 3-6 条 bullet 的直接回应，末尾追加：
  - `Missing Evidence (Author TODO):` 列表（每项一行，`[[TBD: ...]]`）。
- **时态约束**：只允许声称“我们将加入/将在 camera-ready 加入/我们已在 rebuttal 中澄清”；禁止声称“我们已经修改了投稿 PDF”。
- **字符预算**：生成后必须显示 `len(text)`，超限则进入压缩/重写循环直到合规。

## Tasks（可执行改造计划）

### Task 1: 增加 ICML/OpenReview 专用 prompts（不编造 + 5000 chars）

**Files:**
- Create: `prompts/icml_openreview_rebuttal_writer.yaml`
- Create: `prompts/icml_openreview_followup_writer.yaml`
- Create: `prompts/icml_openreview_compressor.yaml`
- Create: `prompts/icml_openreview_compliance_rewriter.yaml`
- Modify: `tools.py`（增加 prompt name mapping 或直接按文件名加载）

**Step 1: 写入 rebuttal writer prompt（禁止新数字 + 占位符 + 5000 chars）**
- 要求：只使用输入中已有的数字；若 reviewer 要求实验但输入缺失，必须输出 `[[TBD: ...]]`，不得输出任何具体数值。
- 要求：禁止 URL；禁止跨 reviewer 引用；按 ReviewerID/Q 编号组织。

**Step 2: 写入 follow-up writer prompt（短回复 + TODO evidence 列表）**
- 产物：一条可粘贴的 follow-up 回复 + “Missing Evidence” 清单（占位符）。

**Step 3: 写入 compressor prompt（超限压缩）**
- 输入：原草稿 + 超限提示 + 必保留要点列表（例如每个 Q 的一句核心回应）。
- 输出：严格 `<= 5000 characters`。

**Step 4: 写入 compliance rewriter prompt（修复违规项）**
- 输入：草稿 + 违规列表（URL/身份线索/时态）。
- 输出：修复后的合规版本（仍需 <=5000 chars；必要时再走 compressor）。

**Step 5: 自检**
- 手工用一段示例 review 文本验证：prompt 不再生成 `*` 占位数值。

### Task 3: 生成 per-reviewer rebuttal（并强制 5000 chars + 合规护栏）

**Files:**
- Modify: `rebuttal_service.py`（新增 `generate_openreview_rebuttals()`）
- Create: `compliance_guard.py`（扫描/报错/自动重写触发）
- Create: `char_budget.py`（`len()` 计数与压缩循环）
- Modify: `app.py`（结果页展示 per-reviewer 输出 + 字符计数）

**Step 1: 实现合规扫描（不依赖 LLM）**
- 规则（最小集，保守）：
  - URL：`http://`, `https://`, `www.`, 常见短链域（`tinyurl`, `bit.ly`, `t.co` 等）。
  - 身份线索：email 正则；电话正则；`@` 句柄（可选）。
  - 禁止时态：匹配 `we have updated|we revised the paper|we changed the manuscript` 等短语。

**Step 2: 实现字符预算器**
- `ensure_within_limit(text, limit=5000)`：
  - 若 `len(text) <= limit`：返回。
  - 否则调用 compressor prompt 生成压缩版；重复最多 N 次；仍超限则返回错误并在 UI 标红。

**Step 3: 实现 per-reviewer 生成函数**
- 输入：按 reviewer 聚合的 `(questions + agent7 strategies)`。
- 调用：`icml_openreview_rebuttal_writer.yaml` 生成草稿。
- 后处理：`compliance_guard` 扫描；有违规则调用 compliance rewriter；最后走 `char_budget`。
- 输出：`{reviewer_id: rebuttal_text}`。

**Step 4: 更新 UI（结果页）**
- 展示为多个可复制块：`R1`、`R2`、`R3`。
- 每块显示字符数与是否超限。
- 替换现有 “estimated values (*)” 警告为 “[[TBD: ...]] 占位符待作者补全”。

**Step 5: Smoke check**
- 启动：`python app.py --port 8080`
- 用一份小 review 文本验证：
  - 输出不含 URL；
  - 不出现“we updated the paper”；
  - 每块 <= 5000 chars；
  - 缺实验处出现 `[[TBD: ...]]`。

## 执行注意事项（给实现阶段）

- 修改要“可回退”：保留现有默认模式（当前 CVPR/ICLR 风格）作为 `generic`，新增 `icml_openreview` 作为可选模式，避免影响现有用户。
- 所有新 prompt 必须显式写入 “Do not fabricate numbers/results”。对任何未提供的结果统一使用 `[[TBD: ...]]`。
- 合规护栏宁可“误报并让作者手动修”也不要漏报（尤其是 URL/短链/身份线索）。
